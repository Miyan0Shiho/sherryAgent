# Memory & Retrieval 契约

本文件定义平台级 `Memory & Retrieval` 模块，不再沿用“单个记忆系统实现”的叙事。

## 职责边界

负责：

- 当前 Run 的上下文装配
- 长期记忆与证据检索
- 压缩、去重、TTL、冷热分层
- 为 Planner 和 Execution Engine 提供检索结果

不负责：

- 决定任务目标
- 直接操作外部工具
- 代替 Observability 存储全部运行日志

## 设计原则

- 优先保留 Evidence，而不是保留未经验证的自由文本记忆
- 记忆与证据要能区分“事实”“推断”“习惯性规则”
- 检索必须兼容 10x / 100x 扩容，不依赖单索引无限增长

## 存储分层

- 短期上下文：服务单个 Run
- 工作记忆：服务单个 Task 的多轮 Run
- 长期记忆：服务跨 Task、跨仓库复用
- 冷归档：服务历史追溯与低频查询

## 准入原则

- 长期记忆优先写入 `Evidence` 的结构化摘要，不直接写入未经验证的自由文本。
- 任何进入长期记忆的内容都必须能回链到 `evidence_id` 或明确标记“仅工作记忆，不可长期沉淀”。
- 敏感信息、临时凭据、一次性诊断上下文默认只允许进入短期上下文或工作记忆。

## 检索策略

- 混合检索：关键词 + 结构化过滤 + 向量召回
- 重排依据：相关性、时效性、重要性、租户/仓库边界
- 预算约束：检索结果进入上下文前必须过预算裁剪

## 压缩、TTL 与过时信息处理

- 压缩必须保留来源引用，不允许只保留脱离来源的自由摘要。
- TTL 至少分为短期上下文 TTL、工作记忆 TTL、长期记忆复审周期三层。
- 超过 TTL 但仍需保留的内容，应先进入冷层或重新验证后再提升热度。
- 过时信息必须显式降权或剔除，不允许与当前事实并列呈现却不标注时效差异。

## 引用与置信度

- 引用格式必须至少包含 `evidence_id`、`source_ref` 和时间信息。
- 结论必须区分“事实引用”与“基于事实的推断”。
- `confidence` 至少分为 `high | medium | low`，低置信内容不得伪装成最终结论。

## 版本化与隔离

- 长期知识必须支持版本化、过期、替换和回溯。
- 仓库、团队、租户、数据集边界必须作为检索过滤条件，而不是仅作为展示元数据。
- 跨边界复用默认关闭，只有明确允许的共享知识才能进入跨仓库或跨团队读取。

## 扩容要求

- 10x：独立索引层 + 热点缓存
- 100x：按仓库/团队/租户分片，冷热分层和异步归档
