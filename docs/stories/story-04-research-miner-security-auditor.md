# Story-04: Research Miner + Security Auditor（调研流水线 + 安全审计）

> 角色说明：本文档是 **验收与演示套件**，用于验证批量调研与安全审计链路，不承担顶层项目规划职责。

## Goal

让 Agent 能“长跑”做调研/分析与安全审计，并且做到：
- 产出结构化报告（可追溯来源、可复现结论）
- 结论有不确定性标注（置信度）
- 安全扫描规则可配置、可审计、可回放

## Non-Goals

- 不把“生成一篇看起来像的报告”当成功，必须满足证据与引用契约
- 不自动处理敏感数据（必须脱敏/最小化）
- 不做大规模破坏性扫描（必须有预算与范围）

## 演示脚本

### 手动交互（CLI/TUI）

1. 用户输入一个调研主题或安全审计范围（例如：依赖风险、CVE、密钥泄露扫描策略）。
2. Agent 输出：研究问题拆解 + 数据来源计划 + 预算 + 输出契约。
3. 执行一次最小样本：产出报告骨架与证据清单。

### 条件触发/后台（Heartbeat/Cron）

1. 每周固定时间触发“调研更新/安全周报”。
2. 自动生成报告并登记行动项（需要确认的高风险动作不自动执行）。

## Output Contract

产出一个 “Research+Security Report”，至少包含：
- `scope`（范围与排除项）
- `sources`（来源列表，带可定位引用；无法引用时必须标注）
- `findings`（发现列表：事实/推断分离）
- `confidence`（每条结论的置信度）
- `risks`（风险清单与建议）
- `actions`（可执行的行动项，带风险等级与所需确认）

## 风险分级与权限策略（写死）

- **LOW**：读 repo、读依赖列表、读配置。自动放行 + 审计。
- **MEDIUM**：写报告文件、追加审计记录。自动放行 + 审计。
- **HIGH**：对外抓取数据、执行外部命令、扫描敏感目录。必须确认 + 限定范围。
- **CRITICAL**：任何会修改系统安全策略/删除文件/提交变更的动作。默认拒绝。

## 失败模式与降级

- 来源不可得：报告必须输出“证据缺口”，不强行下结论。
- 低能力模型幻觉：强制“引用/证据”字段；不满足即判定输出不合格。
- 范围过大：先采样 + 分段执行 + 限流 + 预算硬停止。

## 六层映射

- 交互层：输入调研问题/扫描范围，查看报告与行动项。
- 编排层：拆成“收集 -> 提取 -> 归纳 -> 审计 -> 汇总”流水线。
- 执行层：读并发、写串行；所有行为事件化。
- 自主运行层：定期报告与自检。
- 记忆层：沉淀“调研结论/审计规则/历史问题”（去敏）以减少重复劳动。
- 基础设施层：权限管道确保长跑自动化不越界。
