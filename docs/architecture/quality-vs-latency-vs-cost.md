# Quality vs Latency vs Cost

## 默认原则

系统默认 **quality-first**，但不是无上限堆资源。优化策略必须显式选择，不允许隐式漂移。

## 决策矩阵

| 任务类型 | 默认优先级 | 允许牺牲什么 | 典型策略 |
|------|------|------|------|
| 高风险、高价值、低频 | 质量与安全 | 延迟、部分成本 | 更强规划、更强 review、更严格确认 |
| 中风险、重复性 | 成本与稳定性 | 部分深度 | 模板化、缓存、规则优先、低风险自动化 |
| 高频批量 | 吞吐与预算 | 实时性、局部精度 | 分片、采样、聚合、异步报告 |
| 发布与治理 | 一致性与可回滚 | 延迟、局部吞吐 | 强门禁、强证据、严格决策记录 |

## 预算档位

### `strict`

- 适用：后台、批量、重复任务
- 策略：限制轮次、限制模型升级、优先命中缓存、必要时提前停止

### `balanced`

- 适用：常规交互与团队开发任务
- 策略：允许中等规划深度和有限升级

### `premium`

- 适用：高风险决策、复杂 review、事故分诊
- 策略：允许更深拆解、更多证据、更长上下文、更严格自检

## 降级顺序

1. 先降并发
2. 再降模型档位
3. 再降规划深度或样本量
4. 最后停止并请求人工介入

## 成本控制手段

- 模型路由：小模型优先处理模板化或工具化子任务，复杂审查再升级模型。
- 缓存策略：prompt cache、evidence cache、retrieval cache、decision cache。
- 限流策略：按团队、仓库、任务类型、风险等级限流。
- 审计要求：任何降级都必须在 `Decision.policy_basis` 中记录原因。
