# 规范权威与冲突裁决

本项目采用 **`.trae/specs` + `docs/` 双权威**，目的是同时保证“开发可执行”与“系统契约不漂移”。

## 1. 权威矩阵（写死）

| 类别 | 权威来源 | 负责回答的问题 | 强制性 |
|------|----------|----------------|--------|
| 开发执行（做什么/怎么验收/任务如何拆分） | `.trae/specs/*` | 下一步要做什么？验收是什么？任务清单是什么？ | 必须遵守 |
| 系统契约与叙事（长期结构/术语/安全原则/故事口径） | `docs/*` | 系统长期边界是什么？安全红线是什么？Story 要如何讲与如何演示？ | 必须遵守 |
| 方法论参考（可选模板/写法） | `docs/guides/openspec-guide.md` | 如何写 proposal/design/tasks/spec 的结构化文档？ | 仅参考 |

## 2. 冲突裁决（写死）

发现 `.trae/specs` 与 `docs/` 不一致时：

1. **先分类冲突类型**
   - A. Roadmap/任务/验收冲突（开发执行层面）
   - B. 系统契约/安全原则/术语冲突（长期口径层面）
2. **裁决规则**
   - A 类冲突：以 `.trae/specs` 为准，但必须在 `docs/` 补一条“差异记录”（记录原因与影响）。
   - B 类冲突：以 `docs/` 为准，必须回写 `.trae/specs` 修正后才允许继续开发。

## 3. Definition of Done（文档版 DoD）

任何非 trivial 变更（新增 Story、修改系统契约、调整安全策略、改变任务流程）完成时，至少满足：

- `.trae/specs`：对应 spec/tasks/checklist 已更新（能指导下一位 agent 继续做）。
- `docs/`：相关 Story 或契约文档已更新，且术语一致。
- `docs/INDEX.md`：能导航到新增/修改的关键文档入口。
- 安全相关变更：必须明确落到“权限管道/审计/确认/沙箱”的某一层，否则视为未完成。

> 注：本 DoD 不要求运行测试（除非后续 `.trae/specs` 明确要求）。
